# Memory Taxonomy & Ledger

This document defines the memory classification system and serves as the durable memory ledger for the framework.

## Memory Taxonomy (6 Classes)

All durable memory entries are classified into one of 6 classes based on scope and lifecycle:

### Class 1: Enterprise Policy Memory
**Scope:** Organization-wide  
**Owner:** Enterprise administrators  
**Write Policy:** Enterprise only (user cannot modify)  
**Lifecycle:** Annual review  
**Use Case:** Organization-mandated policies, compliance requirements, security standards

**Examples:**
- "All deployments must be code-reviewed by 2+ people"
- "API keys must be stored in HashiCorp Vault, never in .env"
- "Incident response: page oncall within 5 minutes"

### Class 2: Project Operating Memory
**Scope:** This project (all collaborators)  
**Owner:** Project maintainer  
**Write Policy:** Requires approval (user proposes, owner approves)  
**Lifecycle:** Quarterly review  
**Use Case:** Shared project conventions, architecture decisions, team agreements

**Examples:**
- "Use React hooks, not class components in this project"
- "All DB migrations must have rollback scripts"
- "Feature branches deleted after merge; never keep stale branches"

### Class 3: Personal Durable Preference Memory
**Scope:** User-specific (single developer)  
**Owner:** User  
**Write Policy:** Requires approval (reflect on personal patterns)  
**Lifecycle:** Bi-annual review  
**Use Case:** Personal working patterns, tool preferences, optimization learnings

**Examples:**
- "I work best when I break large tasks into 2-3 hour chunks"
- "I prefer subagents for exploration; main thread for implementation"
- "I always validate framework changes with validate_framework.py before committing"

### Class 4: Session State Memory
**Scope:** Current session only  
**Owner:** Claude (ephemeral)  
**Write Policy:** Auto-updated during session  
**Lifecycle:** Archived at session end (not persistent)  
**Use Case:** Current task state, active decisions, pending approvals, blockers

**Storage:** `~/.claude/session-state.json` (live)  
**Archive:** `~/.claude/session-logs/YYYY-MM-DDTHH-MM-SSZ.json` (after session end)

**Schema:**
```json
{
  "timestamp": "ISO 8601",
  "project_dir": "string",
  "task": "string (what we're doing)",
  "progress": "string (how far we've come)",
  "decisions": ["array of decisions made"],
  "blockers": ["array of obstacles"],
  "next_steps": ["array of what's next"],
  "active_files": ["files being worked on"],
  "plan_file": "path/to/plan if exists",
  "pending_approvals": ["what needs user sign-off"],
  "compaction_count": 0
}
```

### Class 5: Retrieval/Index Memory
**Scope:** Framework-internal  
**Owner:** Framework curator  
**Write Policy:** Auto-generated by framework  
**Lifecycle:** Auto-expired when referenced file deletes  
**Use Case:** Quick reference indices, navigation aids, search help

**Examples:**
- "All agent definitions in ~/.claude/agents/"
- "All hooks configured in ~/.claude/settings.json"
- "Memory entries by project in ~/.claude/projects/"

### Class 6: Reflective Memory
**Scope:** User-specific  
**Owner:** User  
**Write Policy:** Requires approval (introspection)  
**Lifecycle:** Annual review  
**Use Case:** Lessons learned, anti-patterns to avoid, workflow optimizations

**Examples:**
- "When I skip running tests, I introduce bugs in 60% of cases — always run tests"
- "Distributed systems debugging requires whiteboard sketches, not just code reading"
- "I misunderstood this API 3 times; document its quirks in project memory"

## Memory Entry Schema

Every durable memory entry (Classes 2, 3, 5, 6) is a markdown file in the project memory directory:

**Location:** `~/.claude/projects/{project}/memory/{id}.md`

**File structure:**
```markdown
---
id: memory-0001
class: project-operating
owner: user | project | enterprise
scope: personal | project | organization
source: "conversation transcript" | "decision in meeting" | "documentation"
provenance: "2026-04-08 session link" or "link to source"
sensitivity: low | moderate | high | restricted
approval_state: proposed | approved | rejected | expired
created_at: 2026-04-08
review_at: 2026-07-08
status: active | superseded | under_review | archived
supersedes: null | "memory-NNNN"
---

# Brief Title

Body of the memory entry in markdown. Can include:
- Key decision and why
- Important context for future sessions
- Related files or commands
- When to reconsider this decision

## Examples (optional)
Concrete examples or scenarios where this applies
```

**JSON Schema Validation:**
```json
{
  "type": "object",
  "required": ["id", "class", "owner", "scope", "source", "provenance", "sensitivity", "approval_state", "created_at", "review_at"],
  "properties": {
    "id": {"type": "string", "pattern": "^memory-\\d{4}$"},
    "class": {"enum": ["enterprise-policy", "project-operating", "personal-preference", "session-state", "retrieval-index", "reflective"]},
    "owner": {"enum": ["user", "project", "enterprise"]},
    "scope": {"enum": ["personal", "project", "organization"]},
    "source": {"type": "string"},
    "provenance": {"type": "string"},
    "sensitivity": {"enum": ["low", "moderate", "high", "restricted"]},
    "approval_state": {"enum": ["proposed", "approved", "rejected", "expired"]},
    "created_at": {"type": "string", "format": "date"},
    "review_at": {"type": "string", "format": "date"},
    "status": {"enum": ["active", "superseded", "under_review", "archived"]},
    "supersedes": {"type": ["string", "null"]}
  }
}
```

```yaml
  - Approval 2: What and why

  - Total exchanges: N
```

## Memory Curation Workflow

### Step 1: Identify Candidates (During Session)
- Watch for decisions that will guide future work
- Mark patterns that repeat across sessions
- Note lessons learned that cost time/effort

### Step 2: Propose (At Session End)
```
/governance:memory-proposal
```
This command:
- Extracts candidates from conversation
- Classifies by memory class
- Suggests owner, scope, sensitivity
- Checks for duplicates/conflicts

### Step 3: Review (User's Decision)
User examines proposed entries:
- Is this stable/reusable? (not temporary)
- Is this actionable? (can guide future decisions)
- Is the classification correct?
- Approve or reject each

### Step 4: Write (If Approved)
Approved entries get:
- Assigned unique ID (memory-NNNN)
- Written to `~/.claude/projects/{project}/memory/{id}.md`
- Added to MEMORY.md index
- Timestamped `created_at` and `review_at`

### Step 5: Review Cycle (Quarterly+)
- `/memory-cleanup` runs periodically
- Flags entries approaching `review_at` date
- User re-evaluates: keep? supersede? archive?
- Updates `status` and `review_at`

---

## Approval States

| State | Meaning | Transition |
|-------|---------|-----------|
| **proposed** | Candidate identified, not yet approved | → approved (user says yes) → rejected (user says no) |
| **approved** | User has given thumbs-up, entry is live | → superseded (replaced by newer entry) → under_review (needs re-eval) → archived (deprecated) |
| **rejected** | User declined to persist this entry | → (terminal, can be re-proposed later) |
| **expired** | Review date passed, no action taken | → (terminal, can be recovered if still valuable) |

---

## Sensitivity Levels

| Level | Definition | Storage | Sharing |
|-------|-----------|---------|---------|
| **low** | Publicly shareable (e.g., "use React hooks") | Version control OK | Share freely |
| **moderate** | Project-specific (e.g., "API rate limit is 100 req/min") | Version control with team approval | Share within team |
| **high** | Contains business logic or competitive advantage | Version control restricted | Share on-need-to-know |
| **restricted** | Contains secrets, PII, or compliance-sensitive data | Never version control | Never share; session-state only |

## Ownership Defaults

| Scope | Default Owner | Can Write |
|-------|---------------|-----------| 
| **personal** | User | User only |
| **project** | Project maintainer | Maintainer + user (with approval) |
| **organization** | Enterprise admin | Enterprise admin only |

## Memory Limits & Quotas

- **Per-project cap:** 50 active entries (use `/memory-cleanup` to enforce)
- **MEMORY.md index:** Keep under 200 lines (one entry per line)
- **Entry file size:** Keep under 5 KB (focus on actionable insight, not detail)
- **Stale threshold:** 30+ days without review → flag for audit
- **Archive threshold:** 6+ months superseded → can be deleted

```markdown
# Use Python for all framework hooks, not Bash

## Decision
All framework hooks (automation at lifecycle events) must be written in Python, not Bash.

## Why
- **Portability:** Python works identically across macOS, Linux, Windows
- **Error handling:** Try/except is cleaner than bash trap/set -e
- **Testing:** Python modules easy to unit test; bash scripts are fragile
- **Maintainability:** Type hints, docstrings, standard library functions
- **Performance:** Python startup is negligible for <2s timeout constraints

## When to Apply
- Any new hook script
- Any modification to existing hooks
- Conversion of old bash hooks (phased migration)

## Related
- See ~/. claude/scripts/hooks/ for all implementations
- All hooks use common.py for shared I/O utilities
- Settings.json wires all hooks to Python entry points
```
